# TACTICS_LAYER_SPEC.md

## 1. Introduction

This specification defines the architecture, data models, and invariants for the tactics layer within the EdgeLorD ecosystem. It distinguishes between *proof-state tactics* (semantic transformations) and *editor tactics* (textual rendering of semantic patches), ensuring a hard trust boundary, determinism, and non-blocking parallelism. The goal is to provide a robust framework for interactive theorem proving that integrates seamlessly with the Language Server Protocol (LSP) and the Mac Lane kernel.

## 2. Core Concepts

### 2.1. Proof State

A "proof state" is the complete semantic context of a proof at a given point. While explicit "hole" or "metavariable" types were not immediately obvious in `make_rag_min.sh`, the system implicitly operates on:

*   **Goals (G):** Obligations to be proved. Represented semantically, potentially as a form of `RefuteSlice::Obligation` or similar structured type from `src/refute/slice.rs`. A goal is anchored to a specific location in the source.
*   **Context (Γ):** The set of available hypotheses, definitions, and rules in scope. This aligns with `src/loogle/context.rs::GoalContext` (though that's for lemma ranking) and the `RefuteSlice` concept.
*   **Holes/Metas (H):** Unfilled terms or subproofs. These are represented as specific kernel forms (e.g., `(hole <id> <expected_type>)`) that the elaborator understands. The initial `RAG` suggests these are handled by `comrade_lisp` or `tcb_core`.

### 2.2. SemanticPatch Data Model

A `SemanticPatch` is a structured, serializable, and versioned representation of a semantic transformation applied to a Mac Lane document or proof state. It is the atomic unit of change produced by a proof-state tactic.

```rust
// Hypothetical Rust structure for SemanticPatch
// This structure must be canonically serializable for hashing and determinism.
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct SemanticPatch {
    // Unique identifier for this patch, derived deterministically from its content.
    // E.g., SHA256 of canonical serialization.
    pub id: HashValue,
    // Protocol version for forward/backward compatibility
    pub version: u32,
    // The kind of semantic transformation applied (e.g., "IntroBinding", "ApplyLemma", "FillHole")
    pub kind: PatchKind,
    // The target in the document (e.g., a specific goal, hole, or span).
    pub target_anchor: StableAnchor, // From src/refute/types.rs
    // List of semantic diffs, ordered deterministically.
    pub changes: BoundedList<SemanticChange>, // Using BoundedList from src/refute/types.rs for consistency
    // Optional: new subgoals generated by the tactic, ordered deterministically.
    pub new_subgoals: BoundedList<Goal>, // Hypothetical Goal struct
    // Optional: UI metadata for LSP, not affecting semantic content
    pub ui_metadata: Option<UIMetadata>,
    // Proof of soundness if applicable and already checked (e.g., a hash of a certified proof term).
    pub soundness_proof_artifact: Option<HashValue>,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum PatchKind {
    IntroBinding { name: String, type_expr: String },
    FillHole { hole_id: String, term: String }, // term is a Mac Lane surface form string
    ApplyRule { rule_id: String, instantiation: BTreeMap<String, String> }, // Rule name and instantiation of metavars
    Rewrite { rule_id: String, direction: RewriteDirection, target_span: ByteSpan },
    // Add other tactic-specific patch kinds as needed
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum SemanticChange {
    // For now, focus on insertions of new Mac Lane forms.
    // Later: more general semantic diffs (e.g., replacing a sub-expression).
    InsertForm { parent_anchor: StableAnchor, position: InsertPosition, new_form_text: String },
    // A semantic delete of an artifact.
    DeleteForm { target_anchor: StableAnchor },
    // Update an existing form, e.g., type annotation.
    UpdateForm { target_anchor: StableAnchor, new_form_text: String },
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum RewriteDirection {
    LeftToRight,
    RightToLeft,
    Bidirectional,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum InsertPosition {
    Prepend,
    Append,
    Before(StableAnchor),
    After(StableAnchor),
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct Goal {
    pub id: String, // Deterministic ID for the goal
    pub anchor: StableAnchor, // Location in the document that implies this goal
    pub statement: String, // Mac Lane surface form of the goal statement
    pub context: BoundedList<ContextItem>, // Relevant context items
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum ContextItem {
    Binding { name: String, type_expr: String },
    // ... other context items like hypotheses, definitions
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct UIMetadata {
    pub title: String,
    pub description: Option<String>,
    pub icon: Option<String>,
    // ... other UI hints for the client
}

// Re-using HashValue from the existing codebase (e.g., src/caching.rs)
pub type HashValue = String; // Or actual type from tcb_core or similar.
```

#### Determinism Policy for `SemanticPatch`:

1.  **Content-Derived ID:** The `id` of a `SemanticPatch` MUST be a deterministic hash of its canonical serialized content (excluding the `id` field itself, but including its `version`). This ensures immutability and easy comparison.
2.  **Canonical Serialization:** All components within `SemanticPatch` (e.g., `changes`, `new_subgoals`, `instantiation` maps) MUST be serialized canonically. This implies using `BTreeMap` for maps and `Vec` with stable sorting for lists where order is not semantically defined.
3.  **Stable Anchors:** `StableAnchor` from `src/refute/types.rs` (or its equivalent) MUST be used for all document references, ensuring that changes are targeted semantically, not by volatile text offsets.
4.  **Versioned Schema:** The `version` field allows for schema evolution while maintaining compatibility.
5.  **No Nondeterministic Elements:** `SemanticPatch` MUST NOT contain any elements that introduce nondeterminism (e.g., `HashMap` iteration order, random numbers, timestamps not tied to a specific semantic event).

### 2.3. Proof-State Tactic

A proof-state tactic is a pure function from a proof state (goal, context, holes) to a `SemanticPatch` (or a set of patches) or a deterministic failure.

#### 2.3.1. Input Data Model: `TacticInput`

```rust
// Hypothetical Rust structure for TacticInput
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TacticInput {
    pub current_proof_state: ProofState, // See 2.1
    pub target_hole_id: Option<String>, // If applicable, the specific hole being targeted
    pub target_goal_id: Option<String>, // If applicable, the specific goal being targeted
    pub additional_args: BTreeMap<String, String>, // Tactic-specific arguments (e.g., lemma name, rewrite rule)
    pub current_document_snapshot_hash: HashValue, // Hash of the current document for integrity checks
    pub current_compile_options_hash: HashValue, // Hash of compile options
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ProofState {
    pub goals: BoundedList<Goal>, // All active goals
    pub holes: BoundedList<Hole>, // All active holes (if distinct from goals)
    pub definitions_in_scope: BoundedList<StableAnchor>, // References to available definitions
    pub rules_in_scope: BoundedList<StableAnchor>, // References to available rewrite rules
    pub current_focus_anchor: StableAnchor, // The current focus point (e.g., cursor position)
    pub workspace_snapshot_hash: HashValue, // For cache key generation
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Hole {
    pub id: String, // Deterministic ID
    pub anchor: StableAnchor, // Location in the document
    pub expected_type: Option<String>, // Mac Lane surface form
    pub context: BoundedList<ContextItem>,
}
```

#### 2.3.2. Output Data Model: `TacticResult`

```rust
// Hypothetical Rust structure for TacticResult
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum TacticResult {
    Success {
        patches: BoundedList<SemanticPatch>,
        // If the tactic is a choice point, this lists alternative patches
        // All alternatives must also be deterministic.
        alternatives: BoundedList<SemanticPatch>,
        ui_metadata: Option<UIMetadata>,
    },
    Failure {
        reason: TacticFailureReason,
        message: String,
        diagnostic: Option<Diagnostic>, // LSP Diagnostic
    },
    // For tactics that initiate a long-running process (e.g., complex automation)
    // The SemanticPatch would be delivered later via a separate LSP notification.
    InProgress {
        message: String,
        progress_token: String,
    },
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum TacticFailureReason {
    InputMismatch,      // Tactic not applicable to input
    ContextMismatch,    // Required context not found
    FuelExhausted,      // Deterministic search/automation hit resource limit
    InternalError,      // Bug in tactic implementation
    SoundnessCheckFailed, // Proposed patch was rejected by Stonewall
    NoAlternativeFound, // Deterministic choice had no valid options
    // ...
}
```

#### 2.3.3. Failure Modes:

*   **Deterministic Failure:** Tactics must always fail deterministically for the same input. The `TacticFailureReason` enum provides structured reasons.
*   **Choice Points:** If a tactic can produce multiple valid `SemanticPatch`es (e.g., multiple ways to rewrite a term), it MUST return all of them in a deterministically ordered `alternatives` list. The client (EdgeLorD/editor) can then select one based on UI or user preference. The tactic itself does not make arbitrary choices.
*   **Fuel Exhaustion:** Search-based or automated tactics MUST implement explicit fuel limits (`RefuteLimits` in `src/refute/types.rs` serves as a precedent). Hitting the fuel limit is a deterministic failure, not a nondeterministic timeout.

### 2.4. Editor Tactic

An editor tactic is a function that takes a `SemanticPatch` and the current document model, and produces a set of deterministic text edits. It operates entirely on the textual representation and is conceptually *downstream* of proof-state tactics.

#### 2.4.1. Input: `(SemanticPatch, TextDocument)`

*   **SemanticPatch:** The output from a proof-state tactic.
*   **TextDocument:** The current textual content of the file, including its version.

#### 2.4.2. Output: `(Vec<TextEdit>, Optional<UIHints>)`

*   **Deterministic Text Edits:** A `Vec<lsp_types::TextEdit>` (or equivalent internal representation) that precisely describes the textual changes needed to apply the `SemanticPatch`. These edits MUST be stable across multiple runs for the same `SemanticPatch` and `TextDocument`.
*   **Refusal:** If an editor tactic cannot deterministically translate a `SemanticPatch` into text edits (e.g., due to a semantic-text mismatch, outdated text, or ambiguity), it MUST refuse and signal an error. It MUST NOT attempt to "guess" or perform unsound textual transformations.

### 2.5. Minimum Interface EdgeLorD Needs

EdgeLorD, as the LSP server, acts as an intermediary. It orchestrates requests, manages document state, and translates between LSP types and internal semantic types.

#### 2.5.1. Request: `(LspMethod: "edgelord/applyTactic", Params: ApplyTacticParams)`

```rust
// Hypothetical LSP Request for applying a tactic
// Defined in src/lsp.rs or a new edgelord_lsp_types crate.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ApplyTacticParams {
    pub text_document: TextDocumentIdentifier, // URI and version
    pub position: Position, // Cursor position or range in document
    pub tactic_id: String, // Identifier for the requested tactic (e.g., "intro_binder")
    pub arguments: BTreeMap<String, String>, // Key-value pairs for tactic arguments
    pub target_anchor: Option<StableAnchor>, // Explicit anchor if user selected something
    pub context_focus_id: Option<String>, // If targeting a specific goal/hole in a panel
}

// Response from EdgeLorD to client for edgelord/applyTactic
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ApplyTacticResponse {
    pub success: bool,
    pub message: String,
    pub edits: Option<Vec<TextEdit>>, // Text edits produced by the editor tactic
    pub new_goals: Option<BoundedList<Goal>>, // For UI display
    pub ui_metadata: Option<UIMetadata>,
    pub diagnostics: Option<Vec<Diagnostic>>, // Any new diagnostics from re-check
    pub error_details: Option<TacticFailureReason>,
    pub alternatives: Option<BoundedList<SemanticPatch>>, // If the tactic offers choices
}
```

#### 2.5.2. Workflow:

1.  Client sends `edgelord/applyTactic` request.
2.  EdgeLorD constructs `TacticInput` from document state, position, and arguments.
3.  EdgeLorD calls the relevant proof-state tactic function (e.g., from `madlib/tactics/`).
4.  **Stonewall Enforcement:** Upon receiving `TacticResult::Success` with `SemanticPatch`es:
    *   EdgeLorD (or a dedicated `PatchApplicationService`) performs an internal `Q_CHECK_UNIT_V1` or similar query using the document *as if* the `SemanticPatch` were applied.
    *   The `SemanticPatch` is only considered "sound" if the re-check passes and no new critical diagnostics (errors) are introduced by the *semantic content* of the patch.
    *   If the re-check fails, the `SemanticPatch` is rejected, and `TacticFailureReason::SoundnessCheckFailed` is returned.
    *   The `soundness_proof_artifact` in `SemanticPatch` can be used to optimize this re-check or to retrieve a pre-certified component if the tactic produced one.
5.  If sound, EdgeLorD passes the `SemanticPatch` to the editor tactic for textual rendering.
6.  Editor tactic produces `TextEdit`s.
7.  EdgeLorD constructs `ApplyTacticResponse` with `TextEdit`s, new goals, and any relevant UI metadata or diagnostics, and sends it back to the client.

### 2.6. Trust Boundary Enforcement ("Stonewall")

The "Stonewall" boundary (`src/queries/check_unit.rs` serves as the precedent) is enforced as follows:

*   **Patch Validation:** Every `SemanticPatch` produced by a tactic MUST be subjected to a re-elaboration and type-checking pass by the trusted kernel (via the `Q_CHECK_UNIT_V1` mechanism or similar) *before* being translated into `TextEdit`s or committed to the document state.
*   **Rejection Policy:** If the document, after semantic application of the patch, fails to elaborate or introduces new type errors or unsoundness (`SurfaceError`s from `comrade_lisp` or `tcb_core`), the patch MUST be rejected. The tactic is deemed to have failed, and an error is reported to the client.
*   **No Bypass:** Tactics MUST NOT have direct write access to the kernel's internal proof state that bypasses the elaborator's type checking. All changes MUST be proposed as `SemanticPatch`es that are then validated.

### 2.7. Deterministic Choice Policy

When a tactic can yield multiple valid outcomes (e.g., `alternatives` in `TacticResult::Success`), the choice MUST be deterministic:

1.  **Ordered Alternatives:** Proof-state tactics MUST return alternative `SemanticPatch`es in a stable, canonical order (e.g., lexicographically by patch `id` or by a predefined ranking score). The tactic itself does not decide which alternative is "best" arbitrarily.
2.  **Client Selection:** The EdgeLorD server or the client (editor UI) is responsible for selecting one of the alternatives, potentially based on user interaction, configuration, or a predefined deterministic heuristic (e.g., always pick the first, or pick the one with the smallest number of `TextEdit`s).
3.  **No Hash Map Iteration:** Any internal "search" or "selection" logic within tactics MUST use deterministically ordered data structures (e.g., `BTreeMap`, sorted `Vec`) and algorithms that yield stable results across runs.
4.  **Fuel Limits:** Search-based tactics MUST have explicit, configurable fuel limits. Exhaustion of fuel is a deterministic failure, preventing non-terminating or nondeterministic behavior.

### 2.8. Logging and Benchmarking Hooks

Logging and benchmarking will be integrated via existing mechanisms (`src/caching.rs::CacheStats`, `src/refute/orchestrator.rs`, `src/queries/check_unit.rs`).

*   **Measurement Points:** Key measurement points for tactics will include:
    *   `tactic_invocation_count`
    *   `tactic_success_count`
    *   `tactic_failure_count_by_reason`
    *   `tactic_execution_time_ms` (p50, p95, max)
    *   `semantic_patch_generation_time_ms`
    *   `stonewall_recheck_time_ms`
    *   `editor_tactic_rendering_time_ms`
    *   `semantic_patch_size_bytes`
    *   `text_edit_count`
*   **Integration:** These metrics will be collected by EdgeLorD and exposed via LSP logging or custom LSP notifications, compatible with the existing benchmarking infrastructure (e.g., outputting to CSV).
*   **No Bloat:** Metrics collection MUST be lightweight and not add significant overhead to tactic execution.
