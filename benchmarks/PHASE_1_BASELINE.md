## Executive Summary

This document defines the **benchmark strategy** and **measurement infrastructure** for Phase 1 cache performance. The actual measurements will be captured once the ProofSession is instrumented with timing hooks.

**Key insight**: We need real data before optimizing. Phase 1.2B full implementation (SniperDB memo) and Phase 1.2C (fine-grained deps) should only proceed if the data justifies the integration complexity.

## Measurement Infrastructure

### CSV Format

```
timestamp_ms,scenario,uri,edit_id,phase1_hit,phase1_1_hit,compile_ms,end_to_end_ms,diagnostics_count,bytes_open_docs,cache_entries
```

**Fields**:
- `timestamp_ms`: Wall-clock when edit completed (ms since epoch)
- `scenario`: Edit pattern name (hot_edit, unicode, workspace_churn, cross_file)
- `uri`: File being edited
- `edit_id`: Sequential edit number in scenario
- `phase1_hit`: Y/N - Module snapshot cache hit (file_id + content_hash + options_fp + deps_fp)
- `phase1_1_hit`: Y/N - Workspace-aware cache hit (5-component CacheKey)
- `compile_ms`: Duration of `workspace.did_change()` compilation (0 if cache hit)
- `end_to_end_ms`: Total latency from edit enqueue to diagnostics published
- `diagnostics_count`: Number of diagnostics in compilation output
- `bytes_open_docs`: Total size of all open documents (for memory tracking)
- `cache_entries`: "phase1_entries/phase1_1_entries" - current cache sizes

## Benchmark Scenarios

### 1. Hot Edit (Single File, Repeated Changes)

**Purpose**: Measure cache effectiveness for typical editing

**Pattern**:
- Open one file with initial content
- Apply 100 rapid edits with small textual changes
- Each edit cycles through 5 content variants

**Expected behavior**:
- Phase 1 hits (module snapshot) on repeated edits with same content
- Phase 1.1 hits when workspace hasn't changed
- False misses on workspace churn (conservative model)

**Acceptance**: >60% combined hit rate (Phase 1 + Phase 1.1)

### 2. Unicode Worst-Case

**Purpose**: Stress span conversion path and multi-byte handling

**Pattern**:
- Open file with CJK characters: `(def 名前 (hole 値))`
- Edit identifiers and hole names across multi-byte boundaries
- 50 edit cycles

**Expected behavior**:
- Cache hits should not cause latency regressions compared to unicode-naive benchmarks
- P95 latency should remain <100ms

**Acceptance**: No latency regression vs hot-edit scenario

### 3. Workspace Churn (Document Invalidation Cost)

**Purpose**: Measure impact of conservative (all-invalidating) dependency model

**Pattern**:
- Open 5 files, edit them in round-robin
- Track recompilation cost when edits affect unrelated files
- 50 edit cycles

**Expected behavior**:
- Phase 1.1 cache misses when other files change (conservative model)
- Demonstrates false misses from Phase 1.2C fine-grained deps optimization
- If hit rate >40%, Phase 1.2C may not be worth the complexity

**Acceptance**: Establishes cost baseline for Phase 1.2C ROI calculation

### 4. Cross-File Edit (Dependency Impact)

**Purpose**: Measure transitive dependency invalidation

**Pattern**:
- File A imports File B imports File C
- Edit File C
- Verify File A cache invalidates (dependency_fingerprint changes)
- Edit File C back to original
- Verify File A cache can re-hit with same content

**Expected behavior**:
- Edit C → workspace_snapshot_hash changes → A cache misses ✓
- Edit C back → workspace_snapshot_hash matches → A cache hits ✓

**Acceptance**: Transitive invalidation works correctly (soundness guarantee)

## Success Criteria

### P50/P95 Latency (Diagnostics Responsiveness)

| Scenario | Target P50 | Target P95 |
|----------|-----------|-----------|
| Hot Edit | <50ms | <100ms |
| Unicode | <60ms | <120ms |
| Workspace Churn | <80ms | <150ms |

If P95 >200ms on any scenario, investigate compilation cost vs cache overhead.

### Cache Hit Rates

| Scenario | Phase 1 Target | Phase 1.1 Target | Combined Target |
|----------|----------------|------------------|-----------------|
| Hot Edit | >70% | >50% | >60% |
| Unicode | >60% | >40% | >50% |
| Workspace Churn | >30% | >20% | >25% |

If combined <25%, Phase 1.2B ROI is questionable (SniperDB memo overhead may exceed benefit).

### Memory Growth

- Baseline (0 edits): ~10MB (empty caches)
- After 1000 edits: <200MB (cache bounded to 1000 entries)
- P95 memory spike: <500MB

## Data Analysis Plan

Once measurements are collected:

1. **Import CSV into pandas/spreadsheet**
2. **Calculate aggregate statistics**:
   - Hit rate: sum(hits) / (hits + misses)
   - P50/P95 latency: sort(end_to_end_ms), select percentiles
   - Compiles avoided: count(phase1_hit OR phase1_1_hit)
   - Memory growth: max(cache_entries) across time

3. **Compare scenarios**:
   - Hot edit vs workspace churn: estimate false-miss cost
   - If workspace churn hit rate <30%, Phase 1.2C is justified
   - If hot edit hit rate >70%, SniperDB memo (Phase 1.2B) may be overkill

4. **Decision criteria**:
   - **Proceed with Phase 1.2B** if: combined hit rate ≥50% AND no latency regression
   - **Skip Phase 1.2B, go to 1.2C** if: combined hit rate <40% AND workspace churn shows >50% false misses
   - **Re-architect** if: hit rate <25% (caching not working as designed)

## Why Measure Now?

**The blind optimization problem**: Without data, we might spend weeks on Phase 1.2B (SniperDB memo integration) only to discover that:
- The conservative dependency model's false misses are acceptable (no need for 1.2C)
- SniperDB persistence overhead exceeds benefit
- Compilation is already fast enough

**Measuring first unlocks**:
- Confident ROI justification for Phase 1.2B/C
- Quantified performance targets for review
- Evidence-based architectural decisions

**Goal**: Ship the minimal set of optimizations that achieves >60% hit rate with <100ms P95 latency. If Phase 1 already does that, both Phase 1.2B and 1.2C can be deferred indefinitely.
